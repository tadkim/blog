SVG study

TED 프로젝트 진행 중에 SVG를 만드는 방법, 콘트롤 하는 방법을 잘 알아야 지속가능하게 콘텐츠를 만들 수 있음을 깨닫고 공부를 시작하게 되었다.

SVG란

SVG는 Scalable Vector Graphics이다. 웹에서 사용하는 vector에 기반한 그래픽을 선언하는데 쓰인다. XML 형식을 선언한다. SVG는 확대되거나 사이즈 조절을 해도 품질이 나빠지지 않는다. SVG 파일의 각각의 요소들과 속성들은 애니메이션이 될 수 있다. SVG는 W3C 권고안을 따른다. SVG를 다루려면 HTML과 기본적인 XML에 대한 지식이 있어야 한다.

XML이란

XML은 EXtensible Markup Language이다. 늘릴 수 있는 마크업 언어라는 것이다. XML은 데이터를 저장하고 운송하기 위해 고안되었다. XML은 데이터 운송 뿐만 아니라 데이터를 공유, 사용하는 것도 쉽게 해준다. XML은 사람과 기계 양쪽 모두 읽을 수 있는 형태로 만들어졌다. 다음 예시를 보면 무슨 말인지 이해할 수 있다.

XML과 HTML의 차이점

XML과 HTML은 다른 목적으로 제작되었다.

HTML이 데이터를 보여주는 데 초점을 맞춘 것이라면 XML은 데이터를 운반하는 데 중점을 둔다. XML은 정보가 어떻게 보여질 지에 대한 정보는 포함되지 않는다. 따라서 동일한 XML 데이터가 다양하게 보여질 수 있다. XML은 데이터를 HTML에서 분리시켜준다. 따라서 데이터를 수정할 때 HTML이 아니라 XML을 고치면 된다. 몇 줄의 자바스크립트 코드로 XML 파일을 읽어들일 수 있고 HTML 페이지의 데이터를 업데이트할 수 있다.

XML 태그는 p,body,h1 등 사전 정의되어 있는 HTML 태그와는 달리 사전 정의되어 있지 않다. HTML에서는 닫는 태그를 사용하지 않아도 제대로 작동하는 경우가 있지만 XML은 반드시 닫는 태그를 써주어야 한다. <br />도 마찬가지. 그리고 XML은 대소문자 구별을 한다. 따라서 <Message>라는 태그에 </message>라는 닫는 태그를 쓰면 안된다. 또한 속성값은 반드시 따옴표 안에 위치해야한다. 그리고 < (여는 괄호)는 XML에서 새로운 요소의 시작으로 인식하기 때문에, 반드시 &lt;로 치환해주고, &또한 &amp;로 치환해주어야 한다.

필수적인 것은 아니지만 >,',"도 >는 &gt;로, '는 &apos;로, "는 &quot;로 치환해주는 습관은 바람직하다.

주석을 적는 데 있어서 주석 중간에 연속된 두 개의 dash(--)를 넣으면 안된다. 단 중간에 공백을 낀 두 개의 dash(- -)는 가능하다.

HTML이 여러 개의 공백을 하나로 처리하는 것과는 달리 XML은 공백을 있는 갯수대로 보존한다.

기타 XML 규칙들

XML 엘리먼트는 대소문자를 구별하며, 문자 혹은 _로 시작되어야 한다. XML 엘리먼트 이름은 xml이라는 문자를 포함하는 것으로 시작되면 안된다.

XML Attributes

정보를 표기할 때 Elements를 사용하는 것이나 Attributes를 사용하는 것이나 결과는 다르지 않다. 또한 Elements와는 달리 Attributes는 복수의 값을 포함할 수 없고, 트리 구조를 포함할 수도 없어서 추후에 수정이 어렵다는 단점이 있다. 그럼에도 불구하고 Attributes를 사용해야하는 경우는 Metadata (data about data)를 나타낼 때이다. Elements의 자체적인 속성이라기 보다는 식별을 위한 데이터를 나타낼 때 사용하면 좋다는 뜻이다.

XML에서 Extensible의 의미는?

특정 태그가 없어져도, 추가되어도 괜찮다는 의미.

XML에서 xmlns의 의미 (Namespace)

xml은 사용자가 원하는 태그를 만들어 사용할 수 있는 뛰어난 확장성을 가지고 있다. 그러나 이러한 xml의 확장성은 같은 형식의, 혹은 같은 의미의 문서와 태그들을 여러 분야에서, 여러 업체에서, 또는 다수의 개인이 끊임없이 만들어가고 있다는 큰 문제점이 있다. 새롭게 정의된 수많은 문서와 태그들을 사용하는 과정 속에서 이름이 충돌되거나 같은 의미를 다른 이름으로 표현하는 등의 부작용이 발생하는 것이다. 이런 충돌을 막기 위해 Namespace를 지정한다.

Namespace란 XML 문서에서 Element나 Attribute의 이름으로 쓰일 수 잇는 이름들의 집합이다. 즉, 논리적인 이름 구분을 위해 사용된다. 파일 시스템에 비유한다면 개별적인 이름은 파일이고 이름 공간은 디렉터리라고 이해할 수 있다. XML에서는 Namespace를 URI로 식별한다. URI는 인터넷 전체에서 고유한 값으로 사용된다. 그러므로 URI로 식별되는 Namespace와 Namespace 안에서 정의되는 이름들도 모두 인터넷 전체에서 유일하다.

예를 들어 ‘class’라는 이름의 Element가 어떤 학교의 강의를 뜻하는 class로, 또는 항공기 좌석 등급을 뜻하는 class로, 때로는 특정 그룹에서 수준을 평가하는 class로 사용된다고 하면 같은 Element 식별자인 ‘class’이지만 어느 부분에서 사용되느냐에 따라 의미가 달라질 수 있다.

그것들을 구분하려면 각 각을 구분할 수 있는 URI와 연결시켜야 한다. 학교의 강의를 뜻하는 class는 http://www.sunacademy.ac.kr, 항공기 좌석 등급을 뜻하는 class는 http://www.koair.co.kr , 특정 그룹의 수준을 구분하는 class는 http://www.level.co.kr 과 같이 URI와 연결시켜 각 각을 고유의 의미로 사용할 수 있어야 한다. <class namespace=” http://www.sunacademy.ac.kr”>라고 사용하면 여기에서 사용되고 있는 class가 학교 강의를 의미한다는 것을 누구나 인지할 수 있는 것이다.

XML Namespace를 활용하기 위해서는 우선 Namespace를 사용한다는 것을 선언해야 한다. XML 문서에서는 하나의 Namespace URI가 하나의 접두어(prefix) 형태로 대표되어 사용된다. xmlns는 예약되어 있는 키워드로 Namespace를 정의한다는 의미이며, 접두어는 XML 태그 이름에 적용되는 규칙을 그대로 따른다. 다만 대소문자 상관없이 XML이라는 문자열로 사용할 수는 없다. 접두어는 Namespace의 URI를 대신하는 대표 이름으로 이해하시면 된다. 접두어를 잘 정하는 것은 태그 이름을 정하는 것 만큼이나 중요하다. 지정한 XML Namespace의 의미를 잘 담을 수 있는 이름으로 정해야 한다.

XML Namespace로 URI를 지정하도록 되어 있지만 실제 지정된 URI에 물리적으로 꼭 무엇인가 저장되어야 하는 것은 아니다. 다시 말해 물리적으로 어떤 문서나 파일이 존재할 수도 있고 없을 수도 있다.

URI를 따라가면 반드시 정의된 파일이 있는 것이 아니라 URI에서 지정한 곳에서 사용하는 태그 혹은 이미 그 의미와 기능을 정의해 둔 태그들을 가져와서 재사용하겠다는 의미로 해석해야 한다. 즉, Namespace는 논리적인 이름을 구분하기 위한 것이다.

어떤 내부 Element나 Attribute를 Namespace에 연결시킬 때는 그 Namesapce에 대해 정의된 접두어를 그 Element나 Attribute 이름 앞에 붙이면 된다. XML Namespace를 활용한 태그의 이름은 다음과 같이 XML Namespace에 대한 접두어와 구분자인 ‘:’와 실제 태그이름을 지정하는 부분으로 구성된다.

Namespace는 반드시 선언한 후에만 사용할 수 있고, 선언된 Element와 하위 Element에만 적용이 가능합니다. 그래서, 일반적으로 XML Namespace는 문서의 Root Element로 선언되는 경우가 많다. Root Element 이외의 Element에 선언되었다고 해서 잘못된 것은 아니다.

<sunedu:school xmlns:sunedu=http://www.s.ac.kr> s.ac.kr로 묶인 엘리먼트나 어트리뷰트를 sunedu 라는 접두사로 사용한다. XML Element의 태그 이름과 마찬가지로 XML Attribute에서도 XML Namespace를 활용할 수 있다.

Namespace를 적용할 태그 내부에 xmlns:라는 키워드를 활용하여 Namespace를 지정하는 것은 같다. 다만 Namespace가 활용되는 부분이 태그 이름이 아니라 Attribute 이름을 지정할 때 활용된다는 점이 다르다. 그 의미 또한 Element의 Namespace와 마찬가지로 Namespace에서 지정한 곳에서 정의된 Attribute가 활용된다는 것이 명시되는 것이다.

Attribute는 반드시 시작 태그에 정의 됨으로 파서에 따라서 Attribute의 Namespace의 접두어를 명시하지 않아도 시작태그의 Namespace를 활용한 접두어가 있으면 그것을 따르게 해석하는 경우와 그렇지 않은 경우가 있다. 명확하게는 Attribute도 Namespace의 접두어를 표시하는 것이 좋다.

어트리뷰트에 namespace를 선언한 예


<sunedu:school xmlns:sunedu=”http://www.sunacademy.ac.kr” >
<sunedu:class-list>
<sunedu:class sunedu:room=”java cafa”>Java Programming</sunedu:class>
<sunedu:class sunedu:room=”java moca”>XML Fundamental</sunedu:class>
</sunedu:class-list>
</sunedu:school>

위의 예제를 읽는 모든 사람들은 sunedu로 시작하는 모든 Element들 뿐만 아니라 Attribute들도 ”http://www.sunacademy.ac.kr”이라는 Namespace에 속한다는 것을 알 수 있다.

Namespace 사용 목적

Namespace는 논리적인 이름 구분을 위해 사용되는 것으로 다음과 같은 목적으로 사용된다.

  1. 다른 XML에 존재하는 Element와 Attribute의 이름이 겹치는 것을 구별 하기 위해, 이름의 충돌을 피하기 위해 사용한다.
  2. 사용자가 원하는 이름을 재활용할 수 있도록 이름을 그룹으로 묶는 기능을 제공하기 위해 사용합니다.
