2.1 유지보수 가능한 코드 작성

소프트웨어 버그를 고치는데에는 비용이 들고, 이는 시간이 지날수록 증가한다. 얼마간 시간이 흐른 지난 다음에 코드를 다시 들여다 볼 때는 다음과 같은 이유로 시간이 든다

  • 문제를 다시 학습하고 이해하는데 걸리는 시간
  • 이 문제를 해결하는 코드를 이해하는 데 걸리는 시간

무엇보다 코드를 이해하는데 걸리는 시간을 줄여야한다. 자신이 이전에 짜두었던 코드든, 타인이 작성한 코드든 우리는 여기에 들이는 시간적, 정신적 부담을 줄여야한다. 사업 측면에서도 그렇지만 행복에 있어서도 이는 핵심적인 문제다. 우리 모두는 레거시 코드를 유지보수하느라 몇날 며칠을 쓰기보다는 새롭고 흥미로운 문언가를 개발하고 싶기 때문이다.

또 하나의 진리. 코드는 작성하는 것보다 읽는 데 더 많은 시간이 소요된다."

어떤 문제에 집중해서 깊이 몰입해 이싸면, 앉은 자리에서 한나절만에 상당한 양의 코드를 만들어낼 수도 있다. 하지만 이 코드가 그때 그 자리에서 잘 돌아갔을지 몰라도, 애플리케이션이 완성되는 과정에서 여러가지 일이 생기면서 재검토되고 수저오가 변형을 거치게 된다. 예를 들면 다음과 같은 일들 말이다.

  • 버그가 발견된다.
  • 애플리케이션에 새로운 기능이 추가된다.
  • 애플리케이션이 새로운 환경에서 동작해야 한다.(예컨대 새로운 브라우저가 출시되었다.)
  • 코드의 용도가 변경된다.
  • 코드를 처음부터 완전히 재작성하게 되거나, 다른 구조 심지어 다른 언어로 옮기게 된다.

이 결과 코드를 작성하는 데는 몇 시간 걸리지 않았지만, 코드를 읽는데는 몇 주일의 시간이 들어간다. 코드의유지보수가 애플리케이션의성공에 핵심이 되는 이유가 바로 여기에 있다. 유지보수 가능한 코드란 다음과 같은 특징을 지닌다.

유지보수 가능한 코드의 조건들

  • 읽기 쉽다.
  • 일관적이다.
  • 예측가능하다.
  • 한 사람이 작성한 것처럼 보인다.
  • 문서화되어 있다.
