# CSS 모범적인 개발 방식 - - - ## 읽기 쉬운 스타일시트 작성 작업 기간 동안 코드를 잘 정리하면 실제로 시간을 절약할 수 있다. 1. 소개 및 목차를 작성한다. 1. 색상, 폰트, 기타 상수를 정의한다. 1. 목차의 구조를 반영한 영역 헤더를 사용하여 영역을 구분한다. 1. @를 사용하는 룰은 다른 CSS 보다 상단에 둔다. 1. CSS의 구체성을 활용하여 계층구조를 적용한다. ex)
    article { ... }
    article .intro { ... }
    article .intro figure { ... }
    article .intro p { ... }
    article .intro p em { ... }
    article .intro p:firstline { ... }
- - - ## CSS 라이브러리와 프레임워크 CSS 라이브러리는 웹 사이트 전체 및 여러 웹 사이트에서 사용할 수 있는 공통 스타일 모음을 말한다. 이러한 라이브러리에는 CSS 리셋, 일반 타이포그래피, 일반 트랜지션 스타일, 기타 일관성이 필요한 스타일 요소들이 들어있다. 간단한 웹 사이트에는 프레임워크가 오히려 독이될 수 있지만 중간 규모의 사이트만 하더라도 프레임워크를 통해 시간 절약이 가능하다. Bootstrap, Foundation, Pure, Skeleton, Base Framework, Gumby Framework, Responsive Grid System 등 여러 종류가 있습니다. 자세한 내용은 다음 출처를 참고하세요. ### 참고 문서 [최고의 반응형 CSS 프레임워크 18선 - 2013년 7월 기준](http://gokgo.tistory.com/96) [UI Kit CSS Framework 어떤 것을 사용할 것인가?](http://mobicon.tistory.com/353) - - - ## 스타일시트 전략 단일 페이지가 아닌 사이트에서 여러 스타일을 포함한 CSS 문서들을 관리하는 것은 전략이 필요하다. 페이지 로딩 속도에 영향을 주는 것은 연결해야하는 파일의 갯수와 용량이다. 따라서 서버에서 여러 파일을 불러오게 하지 않기 위해 CSS를 한 파일에 몽땅 집어넣어버릴 수도 있고, 용량이 작은 파일로 나누어 여러번 불러오게 할 수도 있다. 이 둘 사이에서 균형점을 잘 잡아야 한다. 1. 전체 통합 방식
모든 CSS를 하나의 CSS 파일에 넣고 를 통해 불러오는 것이다. 비추. 2. 분할 정복 방식
먼저 모든 페이지에서 사용할 전역 스타일 파일을 링크하고 이어서 각각의 해당 페이지에서만 사용하는 스타일을 링크한다. 3. 합체 방식
최종적으로 HTML 파일에는 개별 CSS 파일을 `@import`로 몽땅 호출한 합체된 CSS 파일만을 호출한다. 그러나 `@import`를 사용하면 종종 외부 스타일 시트가 동시에 로드되지 못한다는 사실이 알려지면서 쓰이지 않게 되었다. 4. 동적 방식
서버 사이드 코드 작성기술이 필요하므로 개발자와 상의하세요. - - - ## CSS 코드의 유효성 검증 [css-validator](jigsaw.w3.org/css-validator)라는 사이트를 이용해서 사이트의 CSS 코드가 W3C 표준에서 정한 요구사항을 모두 준수하는지 체크해볼 수 있다. 위의 사이트에 들어간 후 검사를 받을 페이지의 CSS 파일 주소를 링크한다. css-validator로 유효성 검사를 하면 다른 브라우저와 호환이 안되는 것은 유효하지 않은 CSS라고 되어 나온다. 꼭 모든 브라우저에서 호환이 되는 CSS 코드를 사용하라는 것은 아니지만, 유효성 검사를 통해서 코드 에러를 찾기가 한층 쉽다. - - - ## CSS 최소화 읽기 쉬운 CSS를 만들기 위해 사용한 수많은 주석, 공백, 줄바꿈은 결국에 파일의 크기를 늘린다. 따라서 사이트를 웹에 배포하기 전에 파일 크기를 줄이고 싶다면 CSS 코드의 최소화 버전을 만들면 된다. 그리고 읽기 쉬운 버전은 수정에 대비하여 꼭 남겨둔다. ### CSS 최소화 방법 1. [Minify CSS](www.minifycss.com/css-compressor)를 방문한다. 2. CSS 코드를 로드하거나 복붙한다. 3. 옵션을 선택하고 변경해가며 원하는 결과가 나올 때까지, 문제가 없는 결과가 나올 때까지 반복한다. (다만, "유효하지 않은 속성이 발견되었다."는 메시지는 CSS3을 사용할 경우에, 최소화툴에서 아직 지원하지 않는 것이 있어서 나타나는 메시지이므로 신경쓰지 않아도 된다.) - - - ## 32가지 모범 개발 방식 책 386p 1. 값이 0인 경우를 제외하고는 항상 단위를 명시한다. 2. HTML 구조를 먼저 잡고 CSS 표현을 나중에 한다. 3. `DOCTYPE`을 지정한다. 4. 모든 스타일은 외부 스타일로 관리한다. 5. 외부 스타일 시트의 수를 최소한으로 유지한다. 6. ``에 스타일 링크를 두고 ``에는 절대 두지 않는다. 7. HTML 문서에 스타일을 추가할 때는 ``를 사용하고 다른 스타일 시트에 스타일을 추가할 때는 `@import`를 사용한다. 8. HTML 엘리먼트의 기본 스타일도 다시 스타일링한다. 9. 값이 아닌 일반적인 클래스명을 사용하여 재사용성을 높인다. 10. 구체적인 ID명을 사용한다. 11. 모든 페이지를 고유하게 식별할 수 있도록 바디 태그나 모든 페이지에 고유 클래스명이나 ID를 추가한다. 12. 공백으로 구분하여 여러 클래스명을 지정하여 분리, 조합을 통해 중복되는 스타일을 지정한다. 13. CSS 속성과 값이 같은 엘리먼트들은 단일 CSS 규칙으로 합친다. 14. 폼 엘리먼트, 버튼, 기타 UI 엘리먼트에는 마우스 오버나 버튼 클릭 등 동적 스타일을 사용한다. 15. 미디어 쿼리 및 `@media` 규칙을 사용해 기기에 특화된 페이지를 만든다. 16. 클래스나 ID도 유용하긴 하지만, 페이지 구조의 유연성을 위해 CSS 구체성을 활용한 선택자를 사용한다. 17. `!important`를 사용하지 않는다. 18. 불필요하고 반복적인 선언을 피한다. 캐스케이딩을 통해 부모에만 스타일을 설정하면 되는 것을 자식에게까지 반복적으로 설정하지 않는다. 19. 폰트 크기는 상대 크기를 사용하여 페이지 크기에 따라 텍스트를 균일하게 조정할 수 있도록 한다. 20. `font`같은 단축 속성을 사용하여 같은 계열의 여러 스타일을 한번에 지정하는 것에 익숙해지자. 21. 색상값은 알파값까지 조절가능하고 어느 색상인지 유추가 쉬운 RGB를 사용하자. 22. 인터페이스 크롬에 배경 이미지나 배경 스타일을 사용한다. 23. CSS 스프라이트를 사용하여 로딩 속도를 줄이고 상태 변경시 깜빡임도 줄이자. 24. 배경 그라이언트는 CSS를 사용하자. 25. CSS 리셋을 활용하여 여러 브라우저에서도 일관적인 디자인이 가능한 깨끗한 상태에서 작업을 시작하자. 26. 패딩은 Internet Explorer나 박스 모델의 문제 탓에 너비나 높이가 지정된 엘리먼트에 문제를 일으킬 수 있다. 선택할 수 있는 것이라면 패딩보다는 마진을 사용하자. 27. 개발시 파이어폭스, 크롬, 사파리에서 코드를 테스트하고 인터넷 익스플로러에 맞게 코드를 수정한다. 28. 모든 링크 상태의 스타일을 지정한다. `:visited`,`:active` 등의 모든 상태에 스타일을 지정하는 게 사용자에게 시각적인 피드백을 주는 방법이다. 29. 주석을 사용해서 메모를 관리하고 코드, 특히 상수를 조직화한다. 30. 코드를 조직화하기 위해 CSS 구체성을 활용한다. 31. `@import`보다 ``를 선호한다. 대부분의 브라우저에서 임포트한 스타일보다 링크한 스타일 시트를 더 빠르게 다운로드한다. 32. 배포 전에는 공백, 줄바꿈, 주석 등을 지워도 상관없으니 CSS를 최소화하자.
